如果你要用 **UE5 + GAS** 做类似守望先锋的第一人称英雄射击游戏，前期最重要的不是马上写技能，而是先确定：**多人网络模型、角色/技能架构、武器架构、伤害流程、数据驱动方式、GAS 的所有权和复制策略**。这些一旦后期推倒会非常痛。

---

## 一、前期必须先确定的核心问题

### 1. 游戏类型边界

你要先明确项目到底是哪种 FPS：

- **纯 FPS 射击**：枪械、命中、换弹、后坐力是核心。
- **英雄 FPS**：每个英雄有独立技能、被动、终极技能、资源系统。
- **类守望先锋**：通常还包括：
  - 英雄切换
  - 队伍阵营
  - 复活/回合/占点/推车
  - 技能冷却
  - 大招能量
  - 状态效果
  - 护盾/护甲/生命值
  - 位移技能
  - 服务器权威多人同步

如果是类守望先锋，建议从一开始就按 **多人英雄 FPS** 架构设计，而不是先做单机 FPS 再改多人。

---

## 二、网络架构必须早定

### 1. 服务器权威

射击游戏建议采用：

- **服务器权威**
  - 客户端负责输入、表现、预测。
  - 服务器负责最终判定：命中、伤害、死亡、技能生效。
- **客户端预测**
  - 移动、部分技能释放、开火反馈需要预测。
- **服务器回滚/延迟补偿**
  - FPS 命中判定后期非常关键。
  - 如果是 hitscan 武器，最好早期就预留延迟补偿架构。

不要让客户端直接决定伤害，否则很难防作弊。

---

### 2. Dedicated Server 还是 Listen Server

前期要确定：

- **正式多人竞技**：推荐 Dedicated Server。
- **本地联机/测试**：可以先 Listen Server。
- **类守望先锋**：最终基本应按 Dedicated Server 设计。

架构上不要依赖本地玩家逻辑，否则后期迁移 Dedicated Server 会很痛。

---

### 3. 网络同步对象划分

常见划分：

- `GameMode`
  - 只在服务器存在。
  - 控制比赛规则、胜负、出生、复活。
- `GameState`
  - 所有人同步。
  - 控制比分、阶段、倒计时、目标点状态。
- `PlayerController`
  - 玩家输入、UI、客户端 RPC。
- `PlayerState`
  - 玩家长期状态。
  - 队伍、分数、英雄选择、GAS 推荐放这里。
- `Character`
  - 当前出生的角色实体。
  - 移动、胶囊体、Mesh、武器挂点、相机。
- `AbilitySystemComponent`
  - GAS 核心组件。
  - 类守望先锋项目推荐放在 `PlayerState`。

## 三、GAS 架构必须早定

### 1. `AbilitySystemComponent` 放在哪里

这是 GAS 项目最重要的早期决策之一。

#### 方案 A：放在 `Character`

适合：

- 单机游戏
- 角色不会频繁死亡/重生
- 不需要英雄切换
- 不需要玩家状态跨 Pawn 保留

缺点：

- 死亡、重生、换英雄时 `ASC` 生命周期复杂。
- 大招能量、玩家属性、冷却等容易丢失或迁移麻烦。

---

#### 方案 B：放在 `PlayerState`

适合：

- 多人游戏
- 英雄 FPS
- 需要重生
- 需要换英雄
- 需要大招能量保留或部分保留
- 需要玩家状态独立于当前角色

**类守望先锋项目推荐这个方案。**

推荐结构：

```text
PlayerState
 ├─ AbilitySystemComponent
 ├─ AttributeSets
 ├─ TeamId
 ├─ HeroId
 ├─ Score
 └─ UltimateCharge

Character
 ├─ Movement
 ├─ Mesh
 ├─ Camera
 ├─ Weapon
 └─ AvatarActor for ASC
```

也就是说：

- `PlayerState` 是 GAS 的 Owner Actor。
- `Character` 是 GAS 的 Avatar Actor。
- 玩家死亡后换一个新的 `Character`，但 `ASC` 仍然在 `PlayerState` 上。

---

### 2. GAS 初始化流程

早期必须统一初始化流程，否则后期会出现大量技能失效、属性不同步、客户端拿不到 ASC 的问题。

推荐流程：

```text
PlayerController Possess Character
        ↓
PlayerState 拥有 ASC
        ↓
Character 调用 InitAbilityActorInfo
        ↓
赋予英雄 AbilitySet
        ↓
初始化 AttributeSet
        ↓
绑定输入
        ↓
刷新 UI
```

需要特别注意：

- 服务端和客户端都要正确初始化 `AbilityActorInfo`。
- `PossessedBy` 和 `OnRep_PlayerState` 都要处理初始化。
- UI 不要直接假设 `ASC` 一定立刻有效。

---

### 3. GAS 复制模式

多人项目需要考虑 `ASC` 的复制模式：

- `Full`
  - 所有 GameplayEffect 都复制。
  - 适合单机或低人数。
- `Mixed`
  - Owner 获得完整信息，其他客户端只获得必要信息。
  - 多人英雄 FPS 推荐。
- `Minimal`
  - 更省带宽，但信息少。
  - 适合 AI 或大量单位。

类守望先锋项目通常推荐：

```cpp
ASC->SetReplicationMode(EGameplayEffectReplicationMode::Mixed);
```

前提是 Owner 链路正确，比如 `PlayerState`、`PlayerController`、`Character` 的所有权关系要清晰。

## 四、GameplayTag 体系必须早设计

GAS 项目非常依赖 `GameplayTag`。前期不规划，后期会变成灾难。

### 1. 技能标签

示例：

```text

Ability.Hero.Tracer.Recall
Ability.Hero.Reinhardt.Charge
Ability.Weapon.PrimaryFire
Ability.Weapon.Reload
```

### 2. 状态标签

```text

State.Stunned
State.Silenced
State.Rooted
State.Invulnerable
State.Aiming
State.Reloading
State.Sprinting
```

### 3. 输入标签

```text

Input.Look
Input.Jump
Input.Fire.Primary
Input.Fire.Secondary
Input.Ability.Skill1
Input.Ability.Skill2
Input.Ability.Ultimate
Input.Reload
```

### 4. 伤害类型标签

```text
Damage.Bullet
Damage.Explosive

```

### 5. 资源标签

```text

```

**建议所有技能、状态、输入、伤害、资源都用 GameplayTag 表达，不要用大量枚举硬编码。**

## 五、属性系统架构

类守望先锋项目建议至少拆这些 `AttributeSet`。

也可能有：

## 六、技能架构

### 1. 技能不要直接写死在角色里

错误做法：

```text
TracerCharacter 里面直接写 Blink、Recall、Ultimate
```

推荐做法：

```text
HeroData
 ├─ AbilitySet
 ├─ AttributeInitData
 ├─ WeaponData
 ├─ Mesh
 ├─ AnimBP
 └─ UIData
```

角色只是承载体，英雄数据决定技能和表现。

---

### 2. 使用 AbilitySet 管理技能

建议设计类似：

```text
HeroAbilitySet
 ├─ GameplayAbilities
 ├─ GameplayEffects
 ├─ AttributeSets
 └─ InputTag Mapping
```

例如：

```text
TracerAbilitySet
 ├─ PrimaryFire
 ├─ Blink
 ├─ Recall
 ├─ PulseBomb
 └─ Passive
```

这样换英雄时只需要：

```text
移除旧 AbilitySet
应用新 AbilitySet
初始化属性
刷新武器和 UI
```

---

### 3. 技能类型分类

建议前期按行为分类：

- **主动技能**
  - 冲刺、传送、护盾、治疗、投掷物。
- **被动技能**
  - 二段跳、自动回血、护甲增强。
- **武器技能**
  - 主武器开火、副武器开火、换弹、瞄准。
- **大招技能**
  - 需要资源，通常有特殊 UI 和广播。
- **状态技能**
  - 眩晕、沉默、减速、禁疗、无敌。

---

### 4. 技能释放流程

建议统一为：

```text
输入
 ↓
InputTag
 ↓
激活 GameplayAbility
 ↓
检查 Cost / Cooldown / 状态标签
 ↓
播放预测表现
 ↓
服务器确认
 ↓
生成 TargetData
 ↓
应用 GameplayEffect
 ↓
同步 GameplayCue
```

不要每个技能各写一套释放逻辑。

## 七、武器系统架构

类守望先锋的武器不要简单做成普通枪械系统，因为不同英雄差异很大。

### 1. 武器需要支持多种模式

至少考虑：

- Hitscan
  - 士兵 76、黑百合类型。
- Projectile
  - 法老之鹰火箭、半藏箭矢类型。
- Beam
  - 查莉娅、莫伊拉类型。
- Melee
  - 莱因哈特锤子类型。
- Area Effect
  - 榴弹、炸弹、范围治疗。
- Lock-on / Assist
  - 自动锁定或辅助瞄准类技能。

---

### 2. 开火也建议做成 GameplayAbility

推荐：

```text
PrimaryFireAbility
SecondaryFireAbility
ReloadAbility
MeleeAbility
```

优点：

- 能直接使用 GAS 的冷却、消耗、标签阻塞。
- 可以统一处理沉默、眩晕、换弹、弹药。
- 能支持预测和服务器校验。

---

### 3. 命中判定架构

需要提前决定：

#### Hitscan

```text
客户端立即表现
服务器验证射线
延迟补偿
应用伤害
```

#### Projectile

```text
服务器生成投射物
客户端预测表现
投射物碰撞
应用伤害
```

#### Beam

```text
持续检测目标
周期性 GameplayEffect
同步特效
```

---

## 八、伤害系统架构

伤害不要直接写成：

```cpp
TargetHealth -= Damage;
```

推荐走 GAS：

```text
GameplayEffect
 ↓
ExecutionCalculation
 ↓
Damage Context
 ↓
护盾/护甲/生命处理
 ↓
死亡判断
 ↓
击杀归属
```

### 1. 伤害上下文需要包含

建议设计自定义 `GameplayEffectContext`，至少携带：

```text
Instigator
Causer
Weapon
Ability
HitResult
DamageType
IsCritical
Headshot
SourceLocation
TargetLocation
```

否则后面做击杀回放、爆头、助攻、伤害来源 UI 会很麻烦。

---

### 2. 伤害流程建议

```text
攻击方 Ability
 ↓
构造 Damage GameplayEffectSpec
 ↓
写入伤害数值、伤害类型、命中信息
 ↓
目标 ASC 接收
 ↓
ExecutionCalculation 计算最终伤害
 ↓
修改 Health / Armor / Shield
 ↓
如果死亡，通知 GameMode / PlayerState
```

## 九、输入架构

UE5 推荐使用 Enhanced Input。

建议结构：

```text
Enhanced Input Action
 ↓
InputTag
 ↓
ASC 激活对应 Ability
```

不要在 `PlayerController` 里直接写：

```text
按 Q 调用 Ultimate()
按 E 调用 Skill1()
```

而是做成数据驱动映射：

```text
Input.Fire.Primary → Ability.Weapon.PrimaryFire
Input.Ability.Skill1 → Ability.Hero.Skill1
Input.Ability.Ultimate → Ability.Hero.Ultimate
```

这样换英雄、换平台、改键都会简单很多。

## 十一、动画架构

FPS 英雄射击通常需要两套表现：

- **第一人称表现**
  - 手臂、武器、开火、换弹、技能动画。
- **第三人称表现**
  - 其他玩家看到的全身动画。
- **服务器逻辑**
  - 不应该依赖动画完成来决定关键结果。

推荐：

```text
GameplayAbility 控制逻辑
GameplayCue 控制特效/音效
AnimMontage 控制动画表现
AnimNotify 只做辅助，不做核心判定
```

不要让核心伤害完全依赖动画通知，否则网络同步会很难控制。

---

## 十二、GameplayCue 架构

GAS 中特效、音效、震动、UI 提示建议使用 `GameplayCue`。

例如：

```text
GameplayCue.Weapon.Fire
GameplayCue.Weapon.Impact.Flesh
GameplayCue.Weapon.Impact.Wall
GameplayCue.Ability.Blink
GameplayCue.Ability.Heal
GameplayCue.Status.Stunned
```

好处：

- 逻辑和表现分离。
- 网络同步方便。
- 技能表现可以由数据驱动。

## 十八、最容易踩坑的地方

### 1. `ASC` 放错位置

如果你未来有死亡、复活、换英雄，`ASC` 放在 `Character` 上会增加复杂度。类守望先锋推荐放 `PlayerState`。

```text
GameplayCue.Character.Death
GameplayCue.Character.Respawn
```

---

## 最优原则

**Tag 是系统之间的协议。**

不要让系统之间互相写死判断：

```cpp
if (bIsReloading)
```

更推荐让系统判断 Tag：

```text
角色是否拥有 State.Reloading
角色是否拥有 State.Dead
Ability 是否被 State.Stunned 阻止
```

# 阶段三：武器系统最小架构

## 推荐设计

武器系统分三层：

```text
WeaponDataAsset：静态配置
WeaponInstance：运行时状态
WeaponComponent：角色身上的武器管理器
```

---

## 1. `WeaponDataAsset`

负责静态数据。

建议包含：

```text
WeaponId
WeaponName
FireMode
Damage
FireRate
MagazineSize
MaxReserveAmmo
ReloadDuration
TraceRange
SpreadAngle
RecoilPattern
DamageGameplayEffect
FireGameplayCue
ImpactGameplayCue
ReloadGameplayCue
```

示例配置：

```text
WeaponName = Rifle
FireMode = Hitscan
Damage = 20
FireRate = 10 发/秒
MagazineSize = 30
ReloadDuration = 1.5 秒
TraceRange = 15000
DamageGameplayEffect = GE_Damage_Bullet
```

---

## 2. `WeaponInstance`

负责运行时状态。

建议包含：

```text
CurrentAmmoInMagazine
CurrentReserveAmmo
bIsReloading
LastFireTime
CurrentSpread
OwningCharacter
OwningASC
WeaponData
```

它负责回答这些问题：

```text
CanFire?
CanReload?
ConsumeAmmo?
StartReload?
FinishReload?
GetFireInterval?
GetDamage?
GetTraceRange?
```

---

## 3. `WeaponComponent`

挂在 `Character` 上，负责当前角色装备什么武器。

建议职责：

```text
CurrentWeaponInstance
EquipWeapon
UnequipWeapon
GetCurrentWeapon
AttachWeaponMesh
NotifyWeaponChanged
```

注意：

- `Character` 不应该直接保存大量武器逻辑。
- `GameplayAbility` 开火时，从 `Character` 或 Avatar 上拿 `WeaponComponent`，再拿 `CurrentWeaponInstance`。
- 后期换英雄、换武器，只需要替换 `WeaponDataAsset`。

# 阶段六：伤害系统 GAS 管线

这是最重要的部分之一。

## 伤害总链路

```text
GA_PrimaryFire
 ↓
服务器 Trace 命中目标
 ↓
找到目标 ASC
 ↓
创建 Damage GameplayEffectSpec
 ↓
写入 SetByCaller Damage
 ↓
写入 DamageType GameplayTag
 ↓
写入 HitResult / Headshot 信息
 ↓
ApplyGameplayEffectSpecToTarget
 ↓
目标 ASC 接收 GE
 ↓
DamageExecutionCalculation 计算最终伤害
 ↓
修改 Health Attribute
 ↓
PostGameplayEffectExecute 检查死亡
```

---

## 1. `GameplayEffect` 设计

建议建立：

```text
GE_Damage
```

它不直接写死数值，而是通过 `SetByCaller` 传入：

```text
Data.Damage
```

也可以额外传：

```text
Data.CriticalMultiplier
Data.HeadshotMultiplier
```

---

## 2. 为什么用 `SetByCaller`

因为不同武器、不同技能伤害不同。

比如：

```text
步枪普通攻击：20
狙击爆头：120
火箭范围伤害：80
近战攻击：35
技能伤害：50
```

如果每个都做一个固定 GE，后面会爆炸。

推荐：

```text
同一个 GE_Damage
不同 Ability 根据武器数据填入 Damage
```

---

## 3. `DamageExecutionCalculation`

第一版可以很简单：

```text
最终伤害 = SetByCaller Damage
Health -= 最终伤害
```

但是结构上预留：

```text
护盾
护甲
减伤
增伤
暴击
爆头
无敌
伤害类型抗性
来源 Buff
目标 Debuff
```

---

## 推荐第一版计算逻辑

```text
输入 Damage
 ↓
如果目标有 State.Invulnerable，伤害 = 0
 ↓
如果是爆头，Damage *= HeadshotMultiplier
 ↓
先不处理护甲护盾
 ↓
Health = Max(Health - Damage, 0)
```

---

## 第二版再扩展

```text
Damage
 ↓
Shield 扣除
 ↓
Armor 减伤并扣除
 ↓
Health 扣除
```

例如：

```text
RemainingDamage = Damage

if Shield > 0:
    Absorb = Min(Shield, RemainingDamage)
    Shield -= Absorb
    RemainingDamage -= Absorb

if Armor > 0:
    ReducedDamage = RemainingDamage * ArmorDamageRatio
    Absorb = Min(Armor, ReducedDamage)
    Armor -= Absorb
    RemainingDamage -= Absorb

Health -= RemainingDamage
```

# 阶段八：死亡与复活流程

## 死亡判断链路

```text
GE_Damage 修改 Health
 ↓
AttributeSet::PostGameplayEffectExecute
 ↓
如果 Health <= 0
 ↓
通知拥有者 ASC / Character
 ↓
添加 State.Dead
 ↓
取消当前 Ability
 ↓
禁用输入
 ↓
播放死亡 GameplayCue
 ↓
通知 GameMode
```

---

## 死亡时建议处理

```text
添加 State.Dead
取消可取消 Ability
阻止新 Ability 激活
停止开火
停止换弹
关闭碰撞或切换碰撞通道
播放死亡表现
隐藏第一人称武器
通知 HUD 显示死亡界面
```

---

## 死亡时不要随便做的事

不要直接清空所有 GameplayEffect。

因为有些东西可能需要保留：

```text
大招能量
部分冷却
玩家状态
队伍信息
分数
击杀统计
```

推荐区分：

```text
死亡后清除的效果
死亡后保留的效果
复活时重新应用的效果
```

可以用 Tag 标记：

```text
Effect.RemoveOnDeath
Effect.PersistAfterDeath
Effect.ApplyOnRespawn
```

---

## 复活链路

```text
GameMode 收到玩家死亡
 ↓
记录击杀者 / 受害者
 ↓
启动复活计时器
 ↓
到时间后选择 SpawnPoint
 ↓
Spawn 新 Character
 ↓
PlayerController Possess 新 Character
 ↓
Character 初始化 ASC Avatar
 ↓
重置 Health / Ammo
 ↓
移除 State.Dead
 ↓
触发 Respawn GameplayCue
 ↓
HUD 恢复
```

---

## 复活后 Attribute 处理

建议：

```text
Health = MaxHealth
Shield = MaxShield
Ammo = MagazineSize
移除 State.Dead
移除 State.Reloading
移除 State.Firing
```

大招能量是否保留要看你的游戏规则。

类守望先锋通常可以：

```text
死亡后大招能量保留
换英雄后大招能量部分保留或清空
```

## 弹药 UI 链路

弹药有两种放法。

### 方案 A：弹药放 `WeaponInstance`

适合大多数 FPS。


### 方案 B：弹药放 `AttributeSet`

```text
 ↓

```

优点：

- GAS 统一处理。
- 技能修改弹药更方便。

## 我的建议

```text

```

如果后期有大量“技能改变弹药”的需求，再把部分弹药资源 GAS 化。

# 阶段十二：最小可实现开发顺序

下面是一套非常具体的执行顺序。

---

## Step 1：补齐 GameplayTag

先加入最小 Tag：

```text
Input.Fire.Primary
Input.Reload

Ability.Weapon.PrimaryFire
Ability.Weapon.Reload

State.Dead
State.Reloading
State.Firing

Damage.Type.Bullet
Damage.Hit.Headshot
Damage.Hit.Body

GameplayCue.Weapon.Fire
GameplayCue.Weapon.Impact
GameplayCue.Weapon.Reload
GameplayCue.Character.Death
```

### 验收标准

- Ability 可以通过 Tag 阻止激活。
- 输入可以通过 InputTag 找到对应 Ability。
- 死亡状态可以阻止开火和换弹。

---

## Step 2：创建 `WeaponDataAsset`

定义一把基础步枪的数据：

```text
Damage = 20
FireRate = 10
MagazineSize = 30
ReserveAmmo = 90
ReloadDuration = 1.5
TraceRange = 15000
SpreadAngle = 1.0
HeadshotMultiplier = 2.0
```

### 验收标准

- 不改 C++ 也能调整伤害、弹匣、射速、换弹时间。
- 后续换武器只换 DataAsset。

---

## Step 3：创建 `WeaponInstance`

负责运行时弹药和状态：

```text
CurrentAmmoInMagazine
CurrentReserveAmmo
LastFireTime
CanFire
ConsumeAmmo
CanReload
StartReload
FinishReload
```

### 验收标准

- 弹药不会小于 0。
- 射速受到 `LastFireTime` 限制。
- 弹匣满时不能换弹。
- 备弹为 0 时不能换弹。

---

## Step 4：创建 `WeaponComponent`

挂在角色上：

```text
CurrentWeaponInstance
EquipWeapon
GetCurrentWeapon
OnWeaponAmmoChanged
```

### 验收标准

- 角色 Spawn 后自动装备默认武器。
- HUD 能拿到当前武器信息。
- 复活后可以重新装备武器。

---

## Step 5：创建 `GA_PrimaryFire`

Ability 配置：

```text
AbilityTag = Ability.Weapon.PrimaryFire
InputTag = Input.Fire.Primary
BlockedTags:
    State.Dead
    State.Reloading
    State.Stunned
```

### `CanActivateAbility`

```text
获取 AvatarActor
获取 WeaponComponent
获取 CurrentWeapon
检查 WeaponInstance::CanFire
检查状态 Tag
返回是否允许
```

### `ActivateAbility`

```text
CommitAbility
 ↓
WeaponInstance::ConsumeAmmo
 ↓
触发 GameplayCue.Weapon.Fire
 ↓
服务器执行 Hitscan Trace
 ↓
如果命中目标，Apply Damage GE
 ↓
EndAbility
```

### 验收标准

- 有弹药时可以开火。
- 没弹药不能开火。
- 换弹中不能开火。
- 死亡后不能开火。
- 客户端有开火反馈。
- 服务器造成伤害。

---

## Step 6：创建 Hitscan Trace 函数

建议独立到武器或 Combat 工具类中。

输入：

```text
StartLocation
AimDirection
TraceRange
SpreadAngle
IgnoredActors
```

输出：

```text
HitResult
TraceEnd
bHit
```

### 推荐 Trace 起点

第一版可以用：

```text
PlayerController Camera Location
PlayerController Control Rotation
```

注意 FPS 中枪口位置和相机位置可能不一致。

推荐流程：

```text
从相机中心 Trace 确定瞄准点
 ↓
从枪口向瞄准点做二次 Trace
 ↓
得到最终 HitResult
```

这样可以减少贴墙穿模和子弹从眼睛发射的问题。

---

## Step 7：创建 `GE_Damage`

使用 `SetByCaller`：

```text
Data.Damage
Data.HeadshotMultiplier
```

并绑定：

```text
DamageExecutionCalculation
```

### 验收标准

- 同一个 `GE_Damage` 可以被不同武器复用。
- 伤害数值来自 WeaponData。
- 爆头倍率可以来自 WeaponData。

---

## Step 8：创建 `DamageExecutionCalculation`

第一版逻辑：

```text
读取 Data.Damage
读取是否爆头
如果目标有 State.Invulnerable，伤害为 0
如果爆头，乘以 HeadshotMultiplier
Health = Clamp(Health - Damage, 0, MaxHealth)
```

### 验收标准

- 命中身体扣基础伤害。
- 命中头部扣更高伤害。
- 无敌状态下不掉血。
- 血量不会低于 0。

---

## Step 9：在 AttributeSet 中处理死亡

推荐链路：

```text
Health 被修改
 ↓
PostGameplayEffectExecute
 ↓
如果 Health <= 0
 ↓
通知 Character / PlayerState / ASC
 ↓
添加 State.Dead
 ↓
通知 GameMode
```

### 验收标准

- 血量为 0 后只触发一次死亡。
- 死亡后不能继续开火。
- 死亡状态所有客户端一致。

---

## Step 10：实现 GameMode 复活

服务端流程：

```text
HandlePlayerDeath(Victim, Killer)
 ↓
记录击杀信息
 ↓
UnPossess 或禁用当前 Character
 ↓
设置复活 Timer
 ↓
选择出生点
 ↓
Spawn 新 Character
 ↓
PlayerController Possess
 ↓
初始化 ASC Avatar
 ↓
重置 Attribute
 ↓
移除 State.Dead
```

### 验收标准

- 死亡后几秒自动复活。
- 复活后血量满。
- 复活后能开火。
- ASC 没有丢失。
- HUD 正常刷新。

---

## Step 11：创建 `GA_Reload`

Ability 配置：

```text
AbilityTag = Ability.Weapon.Reload
InputTag = Input.Reload
ActivationOwnedTags:
    State.Reloading
BlockedTags:
    State.Dead
    State.Stunned
```

### 激活流程

```text
CanReload
 ↓
CommitAbility
 ↓
添加 State.Reloading
 ↓
触发 GameplayCue.Weapon.Reload
 ↓
等待 ReloadDuration
 ↓
服务器调用 WeaponInstance::FinishReload
 ↓
移除 State.Reloading
 ↓
EndAbility
```

### 验收标准

- 弹匣未满且有备弹时可以换弹。
- 换弹期间不能开火。
- 换弹完成后弹匣补满或按备弹数量补充。
- 死亡会打断换弹。

---

## Step 12：绑定 HUD

HUD 监听：

```text
Health Attribute Changed
MaxHealth Attribute Changed
State.Dead Tag Changed
Weapon Ammo Changed
State.Reloading Tag Changed
```

### HUD 显示

```text
血量
弹药
是否换弹
死亡界面
复活倒计时
准星
```

### 验收标准

- 受伤后血条变化。
- 开火后弹药减少。
- 换弹后弹药刷新。
- 死亡后显示死亡 UI。
- 复活后 UI 恢复。

# 推荐类职责划分

## `PlayerState`

负责长期玩家状态：

```text
ASC
AttributeSets
TeamId
HeroId
Score
UltimateCharge
```

不要把当前生命体表现逻辑放这里。

---

## `Character`

负责当前实体：

```text
Camera
Mesh
Movement
WeaponComponent
ASC Avatar 初始化
死亡表现
复活表现
```

不要把具体技能写死在这里。

---

## `PlayerController`

负责：

```text
输入绑定
本地 HUD
客户端视角
本地 RPC 入口
```

不要负责伤害计算。

---

## `WeaponComponent`

负责：

```text
装备武器
当前武器实例
武器切换
通知 HUD 当前武器变化
```

---

## `WeaponInstance`

负责：

```text
弹药
射速
换弹状态
武器运行时数据
```

---

## `GameplayAbility`

负责：

```text
开火流程
换弹流程
技能流程
检查 Cost/Cooldown/Tag
调用武器和伤害管线
```

---

## `GameplayEffect`

负责：

```text
描述伤害、治疗、Buff、Debuff
```

---

## `ExecutionCalculation`

负责：

```text
计算最终伤害
处理护盾/护甲/生命
处理爆头/暴击/减伤
```

---

## `GameplayCue`

负责：

```text
开火表现
命中特效
换弹表现
死亡表现
复活表现
```

# 第一版完成后的验收清单

## 单人 PIE 验收

- **开火**：按左键可以开火。
- **弹药**：开火减少弹药。
- **射速**：不能无限高频开火。
- **换弹**：按 R 正常换弹。
- **阻塞**：换弹中不能开火。
- **伤害**：命中目标会扣血。
- **死亡**：血量为 0 进入死亡。
- **复活**：死亡后自动复活。
- **UI**：血量和弹药正确刷新。

---

## 多人 Listen Server 验收

- **客户端开火**：客户端开火有本地反馈。
- **服务器伤害**：只有服务器应用伤害。
- **血量同步**：所有客户端看到目标血量一致。
- **死亡同步**：死亡状态所有客户端一致。
- **复活同步**：复活后所有客户端正常显示。

---

```text
Dedicated Server
2 Clients
模拟 100ms 延迟
```

重点看：

- **开火反馈是否迟钝**
- **伤害是否重复应用**
- **弹药是否客户端和服务器不同步**
- **换弹是否卡住**
- **死亡后是否还能开火**
- **复活后 ASC 是否重新初始化**
- **HUD 是否丢失绑定**

## 关键设计取舍

### `ASC` 放哪里

继续采用：

```text
PlayerState 持有 ASC
Character 作为 AvatarActor
```

这是类守望先锋项目的最优解。

---

### 开火是否走 Ability

推荐：

```text
开火、换弹、副武器攻击全部走 GameplayAbility
```

这样冷却、消耗、状态阻塞、预测、网络同步都能统一管理。

---

### 弹药放哪里

第一版推荐：

```text
弹药放 WeaponInstance
```

原因：

- FPS 武器弹药强依赖武器实例。
- 多武器切换更清晰。
- 不会让 AttributeSet 过早变臃肿。

但这些仍放 GAS：

```text
Health
MaxHealth
Shield
Armor
UltimateCharge
MoveSpeed
```

---

### 伤害怎么做

必须走：

```text
GameplayEffect + ExecutionCalculation
```

不要直接：

```text
Health -= Damage
```

---

### 表现怎么做

推荐：

```text
GameplayCue
```

不要把特效音效大量写死在 Ability 里。

# 最终建议

你的下一步最优路线是：**先不要做复杂英雄，先实现一个“标准士兵型英雄”的完整战斗纵切**。

最低目标如下：

```text
一个角色
一把 Hitscan Rifle
PrimaryFire Ability
Reload Ability
GE_Damage
DamageExecutionCalculation
Health Attribute
Death / Respawn
基础 HUD
GameplayCue
Dedicated Server 同步测试
```

这条纵切完成后，你后面扩展系统会非常顺：

```text
加 Projectile 武器
加 Beam 武器
加护盾护甲
加英雄技能
加大招
加英雄选择
加比赛目标
加延迟补偿
```

- **最优策略**：先完成 FPS + GAS + 多人同步的最小战斗纵切。
- **核心链路**：输入 → Ability → 武器 → 服务器命中 → Damage GE → Attribute → 死亡复活 → HUD/Cue。
- **关键原则**：服务器权威、客户端预测表现、伤害走 GAS、表现走 GameplayCue、武器数据驱动、状态用 GameplayTag 管理。
